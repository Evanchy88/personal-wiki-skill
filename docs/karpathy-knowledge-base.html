04 / 痛点

无状态 AI 的"失忆症"

❌ 传统 AI 会话

  • 无状态，会话结束即失忆
  • 每次都要重新解释背景
  • 大量 token 用于说明上下文
  • 知识不积累

✅ 有知识库的 AI

  • 持久化，知识存在文件中
  • 直接读取 wiki，数秒内恢复上下文（取决于知识库规模）
  • Token 聚焦于推理和输出
  • 持续增长，越用越聪明

没有知识库的 AI，就像每次都换了一个新助手。

05 / 痛点

人工维护的知识库不可持续

  • 手动打标签、分类、建链接 → 极其耗时
  • 大多数人只使用了笔记软件的 一小部分 功能（手动整理成本过高）
  • Notion、Evernote 等格式专有，数据迁移困难
  • 知识"写进去"但"用起来"的比例极低

💡 Karpathy 的突破

让 LLM 代替人工做整理工作。

"You rarely ever write or edit the wiki manually, it's the domain of the LLM."

⚠️ 但要注意：LLM 整理的结果需要人工校验，不能完全放任。

11 / 对比

RAG vs Karpathy：怎么选？

维度 RAG 方案 Karpathy 方案
基础设施需配置向量数据库（也有 ChromaDB/FAISS 等免费本地方案）纯文本文件
维护成本中（有成熟自动化方案）低（LLM 自动维护）
检索精度高（需 reranking 优化）中高（依赖 LLM 理解力，有幻觉风险）
可审计性低（向量空间是黑盒）高（.md 人类可直接阅读）
响应速度快（毫秒级）中（LLM 推理数秒）
适用规模10 万+ 文档100-1000 文档
最佳场景企业级知识库个人/小团队研究

RAG 适合大规模、高并发、低延迟的企业场景；Karpathy 方案适合个人、小规模、重理解的知识管理。

14 / 实践

最简启动：3 步开始

1️⃣

创建目录

mkdir -p ~/kb/{raw,wiki}

2️⃣

丢入文档

保存 5-10 篇感兴趣的文章到 raw/

3️⃣

让 AI 编译

用 Claude/Cursor 输入编译指令

完整流程（建议 4 步而非 3 步）

  • 步骤 1：创建目录 + Git init
  • 步骤 2：收集原始材料（Web Clipper / markitdown）
  • 步骤 2.5：内容清洗 — 去除广告、导航、重复内容（容易被忽略的关键步骤）
  • 步骤 3：Claude Code / Cursor / Codex 编译 wiki
  • 步骤 4：定期增量更新 + Linting 质量检查
  • 步骤 5：直接对 Agent 提问，基于 wiki 回答

⚠️ 冷启动建议：先积累 10-20 篇高质量文档再开始编译。需要 Python 脚本编写能力和 LLM API 使用经验

15 / 实践

Prompt 模板

你是一个知识编译引擎。读取 raw/ 目录中的原始文档，
编译成结构化 wiki 到 wiki/ 目录。

【约束条件】
• 每次处理不超过 10 个文件（避免超出上下文窗口）
• 摘要长度：每篇 200-300 字
• 概念文章结构：定义 + 核心要点 + 相关概念链接 + 参考来源
• 使用双向链接语法：[[概念名]]
• 信息矛盾时，在文件顶部添加 ⚠️ 冲突标记

【处理流程】
1. 读取 raw/ 目录，列出所有文件
2. 分批处理（每批 10 个文件）
3. 为每批生成摘要和概念文章
4. 更新 index.md 和反向链接
5. 生成编译报告（处理文件数、生成页面数、发现的问题）

【错误处理】
• 无法解析的文件：记录到 error-log.md，跳过
• 超出 token 限制：优先处理最新修改的文件

17 / 风险

风险、局限与不适用场景

风险 说明 应对策略
LLM 幻觉生成的摘要/链接可能不准确定期人工审核，溯源到原文
错误累积长期自动维护导致质量退化Git 版本控制，支持回滚
规模瓶颈超 1000 篇后管理复杂度激增考虑混合方案（RAG + wiki）
API 依赖LLM 服务中断/涨价影响系统本地开源模型备选
知识时效性过时内容未被识别和标注定期 Linting 增加"时效性检查"步骤
认知外包过度依赖 LLM 整理，削弱自身理解保留部分手动笔记，定期人工通读
供应商锁定深度依赖特定 LLM API 的行为模式保持 .md 纯文本格式，模型可切换
错误传播一个错误定义可能污染多篇关联文章反向链接 + 冲突标记 + 人工审核

❌ 不适用

  • 知识库 > 1000 篇文档
  • 需要毫秒级检索
  • 高度动态数据（新闻/股票）
  • 多模态为主（视频/音频）

✅ 最适合

  • 个人/小团队知识管理
  • 技术文档为主
  • 追求可审计性
  • 100-1000 篇文档规模
